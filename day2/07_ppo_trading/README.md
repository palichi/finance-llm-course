# 📁 Day 2 · 07 · PPO 강화학습 + RAG 설명 결합 — 통합 실습

## 이 실습의 전체 구조

```
                    ┌─────────────────────────────┐
                    │   주(主) — PPO 강화학습       │
                    │   "매수/매도/유보"를 결정한다  │
                    └─────────────────────────────┘
                                  │
                                  │ 결정을 내린 뒤
                                  ▼
                    ┌─────────────────────────────┐
                    │   부(副) — RAG (ChromaDB)    │
                    │   "왜 그런 결정인지" 설명을    │
                    │   거들기 위해 호출된다         │
                    └─────────────────────────────┘
                                  │
                                  ▼
                    최종 답변: "매수 / 매도 / 유보 + 근거 설명"
```

**PPO가 두뇌이고, RAG는 그 두뇌의 결정을 설명하는 보조 자료 검색입니다.**
RAG가 먼저 검색해서 PPO에게 알려주는 구조가 아니라, **PPO가 먼저 결론(매수/매도/유보)을 내고, 그 결론을 RAG가 부연 설명**하는 순서입니다. 이 순서를 반대로 착각하지 않도록 본 실습 전체에서 이 위계를 유지합니다.

## 🎯 배경 및 목적

종목별(파일별)로 분리된 일별 OHLCV 데이터를 사용하여, **"현재 시점에서 2일 후 가격이 오를지/내릴지/별 변화 없을지"를 예측하고, 그 예측에 따라 매수/매도/유보를 결정하는 PPO 강화학습 에이전트**를 만듭니다. 이것이 이번 실습의 **본체(主)** 입니다.

이 PPO 에이전트가 결정을 내린 뒤, **RAG(ChromaDB)를 부수적으로 호출**해서 "왜 그런 판단을 했는지" 설명을 덧붙입니다. RAG는 결정을 내리지 않으며, PPO가 이미 내린 결정에 곁들이는 자연어 설명 자료를 찾아오는 역할만 합니다.

```
ppo_ready/{종목코드}.csv
        │
        ▼
PPO 강화학습 (★ 주) ──────────────▶ 매수 / 매도 / 유보 결정
        │
        ▼ (결정이 내려진 뒤에만 호출됨)
ChromaDB 검색 (부) ───────────────▶ "최근 거래량이 늘며 상승 흐름..." 같은 보조 설명
        │
        ▼
최종 답변 = PPO의 결정 + RAG의 설명
```

## 📊 PPO 설계 핵심 요약 (본체)

| 항목                | 값                   |
|---------------------|-------------------|
| 관찰 윈도우(State)   | 직전 20일 OHLCV |
| 윈도우 이동 방식     | 하루씩 이동 (rolling) |
| 한 에피소드 길이     | 20회의 연속된 예측 스텝 |
| 예측 시계            | 2일 후 가격 방향 |
| 행동(Action)        | 0=매수, 1=매도, 2=유보 |
| 보상(맞음/틀림)      | 매수·매도 적중 +10, 오적중 −10 |
| 보상(유보)           | 상승 시 0, 하락 시 −5 |
| 학습 단위            | 종목 1개 = 환경 1개 = 모델 1개 (독립 학습) |

```

- **STEP 2(PPO)** 는 베이스 모델이 없습니다. "주식 데이터를 보고 매수/매도/유보를 판단하는" 사전학습된 모델이 세상에 존재하지 않기 때문에, 신경망을 **처음부터(from scratch)** 학습시킵니다.
- **STEP 6(임베딩)** 은 `BAAI/bge-m3`라는 베이스 모델에서 시작합니다. "한국어 문장의 의미를 벡터로 바꾸는" 능력은 이미 잘 만들어진 모델이 있으니, 그걸 가져와 **주식 도메인에 맞게 미세조정(fine-tuning)** 만 합니다.

비유하면 STEP 2는 "신입사원에게 주식 매매 판단을 처음부터 가르치는 것"이고, STEP 6은 "한국어는 이미 능숙한 사람에게 주식 용어만 추가로 가르치는 것"입니다.

### STEP 2 · [Claude Code 실습 — 주(主)] PPO 환경 + 학습 스크립트 만들기

Claude Code가 실행되면 아래 프롬프트를 **그대로 복사해서 붙여넣기** 하세요:

```
종목별 PPO 주가 방향 예측 강화학습 에이전트를 만들어줘.

배경 및 목적:
  종목별(파일별)로 분리된 일별 OHLCV 데이터를 사용해서
  "현재 시점에서 2일 후 가격이 오를지/내릴지/별 변화 없을지"를 예측하고,
  그 예측에 따라 매수/매도/유보를 결정하는 PPO 강화학습 에이전트를 만들 거야.
  이 에이전트는 추후 RAG 시스템과 결합될 예정이니,
  예측 결과와 판단 근거를 추적 가능한 형태로 남겨줘.

데이터 구조:
  - 데이터는 종목별로 별도 CSV 파일로 분리되어 있어 (예: 005930.csv, 000660.csv, ...)
  - 각 파일은 날짜순으로 정렬된 [date, open, high, low, close, volume] 컬럼을 가져
  - 종목마다 독립된 환경 + 독립된 학습을 수행해줘 (종목 간 데이터를 섞지 않음)

관찰 윈도우(State) 설계:
  - 윈도우 길이: 20일 — 매 예측 시점에 직전 20일치의 OHLCV 데이터를 관찰값(state)으로 사용
  - 윈도우는 한 에피소드 안에서 하루씩 이동(rolling)해줘. 즉:
    - 1번째 예측: 1일~20일 데이터를 보고 → 22일 시점의 방향을 예측
    - 2번째 예측: 2일~21일 데이터를 보고 → 23일 시점의 방향을 예측
    - 이하 동일하게 한 칸씩 이동
  - state shape: (20, 5) → 20일 × [open, high, low, close, volume] (필요 시 정규화 적용)

에피소드 구조:
  - 한 에피소드 = 20회의 연속된 예측 스텝으로 구성해줘
  - 한 종목의 전체 기간 데이터를 여러 개의 20-스텝 에피소드로 분할해서 학습에 사용해줘
    (윈도우 20일 + 예측 20스텝 + 2일 후 확인을 위한 여유분을 고려해서 분할 방식은 네가 합리적으로 정해줘)
  - 각 스텝마다 "현재 시점 → 2일 후 시점"의 가격 변화를 기준으로 보상을 계산해줘

행동(Action) 정의 — 3가지:
  매 스텝에서 에이전트는 "2일 후 가격이 현재보다 오를지/내릴지/별 차이 없을지"를 예측하고,
  그 예측에 따라 다음 3가지 중 하나의 행동을 선택해줘.
  - 0 = 매수(Buy): 2일 후 가격이 현재보다 많이 오를 것이라는 예측
  - 1 = 매도(Sell): 2일 후 가격이 현재보다 많이 내릴 것이라는 예측
  - 2 = 유보(Hold): 2일 후 가격이 현재와 별 차이 없을 것이라는 예측

  "많이 오름/내림"과 "별 변화 없음"을 구분하는 임계값(threshold)이 필요해.
  2일간 가격 변동률이 +α% 이상이면 상승, -α% 이하이면 하락,
  그 사이는 변화 없음으로 정의해줘. 임계값 α는 하이퍼파라미터로 코드 상단에 노출시키고,
  종목별 변동성 통계(예: 표준편차)를 참고해서 초기값을 설정해줘.

보상(Reward) 설계:
  - 실제로 가격이 상승했는데 매수(0) 선택 → 예측 맞음 → +10
  - 실제로 가격이 상승했는데 매도(1) 선택 → 예측 틀림 → -10
  - 실제로 가격이 상승했는데 유보(2) 선택 → 보상 없음 → 0
  - 실제로 가격이 하락했는데 매수(0) 선택 → 예측 틀림 → -10
  - 실제로 가격이 하락했는데 매도(1) 선택 → 예측 맞음 → +10
  - 실제로 가격이 하락했는데 유보(2) 선택 → 손실 회피 실패 → -5
  - 실제로 가격 변화가 거의 없었는데 매수/매도를 선택한 경우는 원래 설계에 명시되어 있지 않으니
    네가 다음 중 하나로 보완해줘 (선택한 옵션을 코드 주석과 실행 로그에 반드시 남겨줘):
      옵션 A) 변화 없음 + 매수/매도 선택 → 작은 음의 보상(예: -2)으로 처리
      옵션 B) 변화 없음을 상승/하락 판정에 포함시켜 단순화 (임계값 밖이면 무조건 상승 또는 하락으로 귀속)
  - 한 에피소드(20스텝) 동안의 보상을 누적해서 episode return으로 기록해줘

PPO 알고리즘 요구사항:
  - 라이브러리: stable-baselines3 의 PPO 사용 (Gymnasium 환경 인터페이스 기반)
  - 환경: gymnasium.Env 를 상속한 커스텀 환경 클래스 StockDirectionEnv 를 종목별로 생성
    - observation_space: Box — shape (20, 5) 또는 평탄화된 (100,)
    - action_space: Discrete(3) — 0=매수, 1=매도, 2=유보
    - step(): 윈도우를 한 칸 이동, 2일 후 실제 가격과 비교해 보상 계산, 20스텝 도달 시 done=True
    - reset(): 새로운 20스텝 에피소드 시작 위치로 윈도우 초기화
  - 종목별로 별도의 PPO 모델 인스턴스를 학습해줘 (가중치 공유 없음 — 종목별 독립 학습 원칙)

만들어줄 파일:
  1. stock_direction_env.py — 위 환경(StockDirectionEnv) 클래스
  2. train_ppo_direction.py
     - 종목 CSV 파일들을 순회하며 종목별로 PPO 학습을 수행
     - 종목별 학습 완료 후 모델을 ./models/{종목코드}_ppo.zip 형태로 저장
     - 학습 중 episode reward 추이를 로그로 출력
     - 학습 완료 후 검증용 구간에서의 행동별 분포(매수/매도/유보 비율)와 적중률(accuracy) 출력
     - 학습/검증 데이터는 시간 순으로 분리해줘 (앞 80% 학습, 뒤 20% 검증) — 미래 데이터가
       학습에 누설되지 않도록
     - 종목 데이터 끝부분에서 "2일 후" 데이터가 없는 구간(예측 불가 구간)은 에피소드 생성 시 제외
     - 가격 변화율 계산의 기준가(현재가)는 윈도우의 마지막 날 종가로 명확히 정의하고 주석으로 남겨줘
     - 종목별 학습 결과(종목코드, 최종 episode return 평균, 적중률)를 요약하는 표를 마지막에 출력

하이퍼파라미터는 전부 코드 상단에 변수로 노출시켜줘:
  윈도우 길이(20), 에피소드 스텝 수(20), 예측 시계(2일), 임계값(α),
  PPO 학습 파라미터(learning rate, n_steps 등)
```

### 📐 STEP 4 결과를 읽는 법

```
적중률(accuracy) = (매수 적중 + 매도 적중 + 유보 적중) / 전체 예측 횟수

- 적중률 33% 근처 → 사실상 무작위 수준 (3개 행동 중 하나를 찍는 것과 비슷)
- 적중률이 꾸준히 33%를 넘기 어렵다면:
    - 임계값 α 가 너무 좁거나 넓은지 확인
    - 윈도우 길이(20일)나 예측 시계(2일)를 조정해보기
    - 종목별 변동성 차이가 큰 경우, 종목마다 다른 α를 적용
```

- **episode return이 학습이 진행될수록 우상향**하는지가 가장 먼저 확인할 신호입니다.
- 매수/매도/유보 비율이 한쪽으로 완전히 쏠려 있다면(예: 99%가 유보), 보상 설계상
  "안전하게 유보만 선택하는 것"이 더 유리하게 설계됐을 가능성이 있습니다 — 보상값을 조정해보세요.
- STEP 4-1에서 직접 본 5건이 STEP 4의 적중률과 비슷한 비율로 맞고 틀리는지 비교해보세요.
  (5건은 샘플이 작아서 우연히 더 좋거나 나쁘게 나올 수 있다는 것도 함께 체감하는 포인트입니다.)

**여기까지가 본체(主) 완성입니다.** 이미 이 PPO 모델만으로도 "매수/매도/유보"를 결정하고, 그 결정을 직접 눈으로 확인할 수 있습니다. 이어지는 STEP 5~7은 이 결정에 "설명"을 덧붙이는 부수 단계입니다.

### STEP 5 · (부수 단계) 왜 RAG가 필요한가

PPO는 "매수"라는 결정은 내리지만, **"왜 매수인지"는 설명하지 못합니다** (신경망 내부의 숫자 계산 결과이기 때문입니다). STEP 4-1에서 본 것처럼 "매수: 적중"이라는 결과는 보이지만, "최근 거래량이 늘며 상승 흐름이라 매수"라는 식의 **자연어 설명은 PPO 혼자서는 만들 수 없습니다.** 이 역할을 ChromaDB(RAG)가 맡습니다.

```
PPO가 못 하는 것                RAG가 하는 것
─────────────────────────────
"왜 매수인지" 자연어 설명         최근 시세/거래량을 자연어 문서로 검색
```

단, **RAG가 PPO보다 먼저 호출되거나, RAG의 검색 결과가 PPO의 결정을 바꾸는 일은 없습니다.** 순서는 항상 "PPO 결정 → RAG 설명 검색 → 결합"입니다.

**Q. GPU가 없는데 가능한가요?**
```

GPU 환경(AWS A10G 등)에서 실행하면 훨씬 빠릅니다.
```

---

> 💡 이 통합 실습의 핵심 포인트
> **PPO가 주(主), RAG는 부(副)** — 이 순서를 기억하세요.
> PPO가 먼저 "매수/매도/유보"를 결정하고, RAG는 그 결정에 자연어 설명을 덧붙일 뿐입니다.
> RAG가 먼저 검색하거나 PPO의 결정을 바꾸는 구조가 아니라는 점이
> 이 시스템 설계에서 가장 중요하게 기억해야 할 위계입니다.
> 그리고 STEP 4-1처럼, **학습은 숫자로 끝나는 게 아니라 직접 눈으로 확인하는 것**까지가
> 한 세트라는 것도 함께 기억하세요.
