## 0. 이 프로젝트가 무엇을 학습하는가

이 모델은 **"매매 정책"**을 학습합니다 — "2일 후 가격이 오를지 내릴지를 맞히는 퀴즈"가
아니라, **"지금 이 지표 상태 + 내 포지션에서 어떤 행동을 해야 결과적으로 내 계좌(자산가치)가
불어나는가"**를 학습하는 구조입니다.

```
오늘 RSI=75, 이격도=112, 골든크로스 5일전, 미보유 상태
   → 모델 입력
   → "매수 62% / 유보 31% / 매도 7%"
   → 가장 높은 확률의 행동 = 지금 추천 행동
```

학습이 끝나면 **오늘 날짜 지표를 넣었을 때 즉시 "지금 매수/매도/유보" 답을 받을 수
있습니다.** 다만 이 답은 과거 데이터에서 학습한 "평균적으로 유리했던 패턴"이며,
항상 옳다는 보장은 없습니다. 확률값 1위와 2위 차이가 작으면(예: 10%p 이내) 신호가
애매하다는 뜻으로 해석해야 합니다.

### 데이터 — 종목 시세 외에 추가로 필요한 것

1. 데이터의 위치 :  day3/01_golden_close/data
2. 데이터의 정보 :  basDt,srtnCd,isinCd,itmsNm,mrktCtg,clpr,vs,fltRt,mkp,hipr,lopr,trqu,trPrc,lstgStCnt,mrktTotAmt
3. 작어벵 필요한 데이터
| 항목    | 필요 여부 |
|---------------------------------|--------------------------------------------------|
| 종목코드/일자/시가/종가/고가/저가  | **필수** (이미 보유)                              |
| 거래량                           | 선택 — 있으면 신호 품질 향상, 없어도 진행 가능     |
| 초기 자본금 / 거래 단위           | 데이터가 아니라 **시뮬레이션 설정값**으로 직접 정의 |
| 거래비용/수수료율                 | 데이터가 아니라 **가정값** (기본 0.2~0.3%로 시작) |
| KOSPI/KOSDAQ 지수, 업종 상대강도  | 선택 — 있으면 보강, 없어도 동작                   |

### 모델은 종목별로 따로 두는가, 하나로 합치는가

**모델은 하나로, 여러 종목 데이터를 합쳐서 학습합니다.** 단, 다음 두 단계는
종목 경계를 절대 섞으면 안 됩니다.

| 단계 | 종목 분리 여부 | 이유 |
|---|---|---|
| ① 지표 계산 (SMA/EMA/RSI/이격도) | **반드시 종목별로 분리** | 종목 경계를 넘어 이동평균을 계산하면 의미 없는 값이 나옴 |
| ② episode 구성 (포지션·손익 시뮬레이션) | **반드시 종목별로 분리** | 삼성전자를 들고 있다가 SK하이닉스 가격으로 손익이 계산되면 안 됨 |
| ③ 모델 학습 (PPO에 episode를 먹이는 단계) | **여러 종목 섞어서 사용** | 모델이 "이 종목이 무엇인지"가 아니라 지표 패턴 자체의 일반 규칙을 배우게 하기 위함 |

종목코드 자체는 state 입력에 포함하지 않습니다 — 그래야 모델이 "RSI 70 넘으면
위험하다" 같은 종목에 무관한 일반 규칙을 학습하고, 학습에 없던 신규 종목에도
같은 모델을 적용할 수 있습니다.

## 1. 전체 아키텍처

```
[종목별 CSV/DB 시세: 종목코드, 일자, 시가, 종가, 고가, 저가]
        │
        ▼
indicators/  — 종목별로 분리하여 계산 (경계 절대 혼합 금지)
  SMA5·20·60, EMA20, 골든/데드크로스, 이격도20, RSI14
        │
        ▼
env/  — 종목별로 분리된 episode 구성, 학습 시에는 여러 종목 episode를 섞어 사용
  TradingEnv(gymnasium.Env)
  state(9차원, lookback 20일) · action Discrete(3) · reward(자산가치 변화 + shaping)
        │
        ▼
train/  — PPO(stable-baselines3) 학습 → models/policy_ppo.zip 저장
        │
        ▼
inference/  — 학습된 모델 로드 → 임의 종목의 "오늘" state 입력 → action + 확률
        │
        ▼
explain/  — 2단계 분리
  1) rule_based.py : 수치 근거 산출 (결정론적, LLM 미사용)
  2) llm_explainer.py : 1)의 수치만 번역하는 자연어 설명 (LLM, 검증 로직 포함)
        │
        ▼
app.py (Streamlit) — 종목 입력 → 예측 + 확률 시각화 + 판단 근거 + 차트
```

### 모델 / 알고리즘 선택

| 레이어 | 선택 | 이유 |
|---|---|---|
| RL 라이브러리 | **stable-baselines3** | 검증된 구현, gymnasium과 바로 호환, 직접 구현 대비 버그 위험 적음 |
| 알고리즘 | **PPO** | 이산 행동(매수/매도/유보)에 안정적, 하이퍼파라미터에 덜 민감해 초기 구축에 적합 |
| 정책망 | **MLP `[64, 64]`로 시작** | state가 이미 SMA/RSI 등으로 가공된 압축 정보라 LSTM의 이점이 제한적. 추후 `policy_kwargs`만 바꿔 LSTM 비교 실험 가능하게 구조화 |

PPO 대안으로 DQN(학습 불안정, replay buffer 필요)이나 SAC(연속 행동에 강함,
지금은 불필요)도 검토했으나, 현재 요구사항(이산 행동 3종, 초기 구축 단계)에는
PPO + MLP가 가장 적합합니다.

## STEP 1. 프로젝트 구조 설계 합의

```
나는 종목별 일자별 주식 시세 데이터를 갖고 있어. 컬럼은:
- 종목코드 (string), 일자 (YYYYMMDD), 시가, 종가, 고가, 저가

이 데이터로 PPO(stable-baselines3) 강화학습 에이전트를 만들어서
"매수 / 매도 / 유보" 3가지 행동을 결정하게 하고 싶어.

핵심 원칙:
- 이 모델은 "2일 후 가격 방향을 맞히는 퀴즈"가 아니라, "지금 포지션과 지표
  상태에서 어떤 행동을 해야 자산가치가 늘어나는가"를 학습하는 매매 정책이야.
- 모델은 하나만 학습하고, 여러 종목의 데이터를 합쳐서 학습 데이터로 사용해.
  단, 지표 계산과 episode(포지션·손익 시뮬레이션) 구성은 반드시 종목별로
  분리해서 계산하고, 종목 경계를 넘어 섞이지 않게 해줘. 학습 단계에서만
  여러 종목의 episode를 섞어서 모델에 입력해.
- state에는 종목코드 자체를 포함하지 않아 — 모델이 지표 패턴의 일반 규칙을
  배우게 하려는 목적이야.

state(9차원, lookback 20일 시퀀스):
SMA5/20/60 이격률, EMA20, 골든크로스 플래그, 데드크로스 플래그,
이격도20(100 중심화), RSI14(정규화), 현재 포지션(보유여부), 미실현손익

reward: 자산가치 변화율 - 거래비용(기본 0.2~0.3%, config화)
  + RSI>70인데 매수 시 페널티(-0.1)
  + 골든크로스 직후 보유 유지 시 보너스(+0.05)
  + 이격도>110인데 신규 매수 시 페널티(-0.1)
  (가중치는 전부 config로 분리, on/off 플래그도 추가)

action: Discrete(3) = 0매도/1유보/2매수

알고리즘: PPO, 정책망은 SB3 기본 MlpPolicy, hidden layer [64,64]로 시작.
추후 LSTM 정책망으로 교체 비교할 수 있도록 policy_kwargs를 분리된 config로 빼줘.

다음 모듈로 나눠서 작업하자:
- data/ : CSV·DB 로딩, 종목코드/종목명 매핑, 종목별 시간 순 정렬
- indicators/ : 기술지표 계산 (종목별 분리 계산)
- env/ : gymnasium TradingEnv (종목별 episode 경계 분리)
- train/ : PPO 학습 — 여러 종목 episode를 섞어서 단일 모델 학습, 체크포인트 저장
- inference/ : 학습된 모델로 임의 종목의 "오늘" 행동 + 확률 예측
- explain/ : 수치 근거 산출(rule_based) + LLM 자연어 설명(llm_explainer), 2단계 분리
- app.py : Streamlit (종목 입력 → 전체 파이프라인 호출 → 결과 표시)

코드 작성 전에 먼저:
1. 폴더/파일 구조
2. 각 모듈의 입출력 인터페이스 (함수 시그니처 수준)
3. train/ 단계에서 "여러 종목 episode를 섞는" 구체적 방식 (예: 종목별로 episode
   리스트를 만든 뒤 shuffle해서 하나의 학습 큐로 합치는 방법 등)
4. 모델(.zip)이 아직 없을 때와 있을 때 Streamlit 앱이 어떻게 다르게 동작해야
   하는지 — 학습은 별도 CLI 스크립트로 분리하고 Streamlit은 추론만 담당할 것
5. 데이터 분할 전략 (시간 기준 train/validation, 종목 내에서 룩어헤드 누설 방지)

을 markdown으로 정리해서 보여줘. 내가 확인하면 순서대로 구현 시작하자.
```

## STEP 2. 기술지표 계산 모듈

```
indicators 모듈을 작성해줘.

요구사항:
1. 입력: 종목코드/일자/시가/종가/고가/저가 컬럼을 가진 pandas DataFrame

   - RSI14: Wilder's smoothing 방식, 분모 0(하락 없는 구간) 예외처리 포함
   - RSI14_norm: (RSI14-50)/50
3. 종목코드별로 그룹화해서 계산하고, 종목 경계를 넘어 지표가 섞이지 않게 할 것
   (이 부분은 단위테스트로 반드시 검증: 두 종목을 이어붙인 가상 데이터를 만들어서
   경계 지점의 지표가 다른 종목 값에 영향받지 않는지 확인)
4. 초기 구간(이동평균 계산 안 되는 첫 N일) NaN 처리는 옵션(drop/backfill)으로
5. docstring + pytest 단위테스트 (RSI는 알려진 예시값으로 검증)

테스트 통과 확인 후 결과 보여줘.
```

## STEP 3. Gym 환경 (종목별 episode 분리)

```
TradingEnv(gymnasium.Env)를 작성해줘.

state 벡터 (9차원, lookback 20일 시퀀스 → shape (20, 9)):
[SMA5_disp, SMA20_disp, SMA60_disp, golden_flag, dead_flag,
 disparity20_centered, RSI14_norm, position, unrealized_pnl]

- position: 보유=1, 미보유=0
- unrealized_pnl: 보유 중일 때 (현재 종가-매수가)/매수가, 미보유면 0

action: Discrete(3) — 0=매도, 1=유보, 2=매수
- 미보유인데 매도 선택, 보유 중인데 매수 선택 등 무효 행동의 처리 방식을 제안해줘
  (무시 / 작은 페널티 등)

거래 제약:
- 매수는 현금 있을 때만, 매도는 보유 주식 있을 때만 유효
- 거래비용 비율 파라미터화 (기본 0.2~0.3%)

핵심 — episode 경계:
- 환경은 반드시 "단일 종목" 데이터로만 구성되어야 함 (여러 종목을 한 episode
  안에서 이어붙이지 말 것)
- reset() 시 어떤 종목, 어떤 시작 시점으로 episode를 구성할지는 외부(train 모듈)
  에서 주입받는 구조로 설계 (종목 리스트를 섞는 로직은 env가 아니라 train에서 담당)

episode 종료: 데이터 끝 도달 또는 자산이 일정 비율 이하로 하락 시 종료

reset(), step() 구현, render()는 텍스트로 현재 상태 출력.
random policy로 여러 episode 굴려보는 스모크 테스트 작성, 특히 종목이 바뀔 때
포지션/손익이 올바르게 초기화되는지 검증.
```

## STEP 4. Reward 셰이핑

```
reward 함수를 작성해줘:

기본 보상: R_t = (V_t - V_{t-1})/V_{t-1} - 거래비용
 (V_t = 현금 + 보유주식 평가액)

셰이핑 추가:
R_t_final = R_t
  - 0.1 * [RSI>70인데 매수한 경우]
  + 0.05 * [골든크로스 직후 매수 포지션 유지하는 경우]
  - 0.1 * [이격도>110인데 신규 매수한 경우]

요구사항:
- 가중치(0.1, 0.05, 0.1, 거래비용율)는 전부 config로 분리, 튜닝 가능하게
- 셰이핑 on/off 플래그 추가 (순수 수익 기반 reward와 비교 실험용)
- 각 셰이핑 항이 episode 동안 몇 번 발동했는지 카운트 로깅
  (에이전트가 지표 규칙만 흉내내고 있는지 점검용)
```

## STEP 5. PPO 학습 (여러 종목 합산 학습)

```
stable-baselines3 PPO로 학습 스크립트를 작성해줘.

핵심 — 여러 종목 episode를 섞어서 단일 모델 학습:
- 종목별로 시간 기준 train/validation 분리 (예: 특정 날짜 이전 train, 이후 validation)
- train 구간 데이터로 종목별 episode를 독립적으로 생성한 뒤, 학습 시점에는
  종목 순서가 섞이도록 구성 (env reset마다 다른 종목이 무작위로 선택되거나,
  DummyVecEnv/SubprocVecEnv로 여러 종목 환경을 동시에 굴리는 방식 등 — 적절한
  방법을 제안하고 선택 이유를 설명해줘)
- 어떤 방식을 택했는지 코드 주석과 실행 로그에 남겨줘

정책망: MlpPolicy, policy_kwargs={"net_arch": [64, 64]}를 config로 분리
(추후 LSTM 정책망으로 쉽게 교체할 수 있게 구조화)

하이퍼파라미터(learning_rate, n_steps, batch_size, gamma 등)는 config/argparse로.

학습 중 validation set 성과(누적수익률, MDD, 샤프비율, 승률)를 주기적으로 평가,
tensorboard 로깅. 가장 좋은 validation 성과의 체크포인트를 별도 저장.

학습 완료 후 train/validation 수익률 곡선을 차트로 저장.

먼저 종목 2~3개로 빠르게 50000 스텝 smoke run을 돌려서 파이프라인이 에러
없이 동작하는지 확인하고, 그 다음 본 학습 설정을 논의하자.
```

## 주의사항

- **데이터 누설 방지**: 지표 계산 시 미래 데이터가 섞이지 않는지(`.rolling()`,
  `.shift()` 사용이 올바른지), train/validation이 시간 기준으로 정확히
  분리됐는지 매 단계 검증
- **거래비용 누락 시 백테스트 왜곡**: 수수료/슬리피지 없이 백테스트하면
  수익률이 비현실적으로 좋게 나옴 — STEP 3, 5에 반영됨
- **LLM 설명의 한계**: STEP 7 검증 로직은 숫자 일치 여부만 잡아주며, "논리적으로
  말이 되는 설명인지"까지 보장하지 않음 — 실서비스 전 사람이 직접 몇 건 검토 필요
- **확률값의 의미**: PPO policy의 행동 확률이 실제 신뢰도와 정확히 일치한다는
  보장은 없음(calibration 문제) — 1위/2위 확률 차이가 작을 때는 신호로 과신하지 말 것
- **이 시스템은 투자 자문이 아닙니다.** 과거 데이터 기반 백테스트 성과가 미래
  수익을 보장하지 않으며, 실거래 적용 전 충분한 out-of-sample 검증과 paper
  trading 기간을 거치는 것을 권장합니다.

```
오늘 RSI=75, 이격도=112, 골든크로스 5일전, 미보유 상태
   → 모델 입력
   → "매수 62% / 유보 31% / 매도 7%"
   → 가장 높은 확률의 행동 = 지금 추천 행동
```

학습이 끝나면 **오늘 날짜 지표를 넣었을 때 즉시 "지금 매수/매도/유보" 답을 받을 수
있습니다.** 다만 이 답은 과거 데이터에서 학습한 "평균적으로 유리했던 패턴"이며,
항상 옳다는 보장은 없습니다. 확률값 1위와 2위 차이가 작으면(예: 10%p 이내) 신호가
애매하다는 뜻으로 해석해야 합니다.

### 데이터 — 종목 시세 외에 추가로 필요한 것

| 항목 | 필요 여부 |
|---|---|
| 종목코드/일자/시가/종가/고가/저가 | **필수** (이미 보유) |
| 거래량 | 선택 — 있으면 신호 품질 향상, 없어도 진행 가능 |
| 초기 자본금 / 거래 단위 | 데이터가 아니라 **시뮬레이션 설정값**으로 직접 정의 |
| 거래비용/수수료율 | 데이터가 아니라 **가정값** (기본 0.2~0.3%로 시작) |
| KOSPI/KOSDAQ 지수, 업종 상대강도 | 선택 — 있으면 보강, 없어도 동작 |

### 모델은 종목별로 따로 두는가, 하나로 합치는가

**모델은 하나로, 여러 종목 데이터를 합쳐서 학습합니다.** 단, 다음 두 단계는
종목 경계를 절대 섞으면 안 됩니다.

| 단계 | 종목 분리 여부 | 이유 |
|---|---|---|
| ① 지표 계산 (SMA/EMA/RSI/이격도) | **반드시 종목별로 분리** | 종목 경계를 넘어 이동평균을 계산하면 의미 없는 값이 나옴 |
| ② episode 구성 (포지션·손익 시뮬레이션) | **반드시 종목별로 분리** | 삼성전자를 들고 있다가 SK하이닉스 가격으로 손익이 계산되면 안 됨 |
| ③ 모델 학습 (PPO에 episode를 먹이는 단계) | **여러 종목 섞어서 사용** | 모델이 "이 종목이 무엇인지"가 아니라 지표 패턴 자체의 일반 규칙을 배우게 하기 위함 |

종목코드 자체는 state 입력에 포함하지 않습니다 — 그래야 모델이 "RSI 70 넘으면
위험하다" 같은 종목에 무관한 일반 규칙을 학습하고, 학습에 없던 신규 종목에도
같은 모델을 적용할 수 있습니다.

## 1. 전체 아키텍처

```
[종목별 CSV/DB 시세: 종목코드, 일자, 시가, 종가, 고가, 저가]
        │
        ▼
indicators/  — 종목별로 분리하여 계산 (경계 절대 혼합 금지)
  SMA5·20·60, EMA20, 골든/데드크로스, 이격도20, RSI14
        │
        ▼
env/  — 종목별로 분리된 episode 구성, 학습 시에는 여러 종목 episode를 섞어 사용
  TradingEnv(gymnasium.Env)
  state(9차원, lookback 20일) · action Discrete(3) · reward(자산가치 변화 + shaping)
        │
        ▼
train/  — PPO(stable-baselines3) 학습 → models/policy_ppo.zip 저장
        │
        ▼
inference/  — 학습된 모델 로드 → 임의 종목의 "오늘" state 입력 → action + 확률
        │
        ▼
explain/  — 2단계 분리
  1) rule_based.py : 수치 근거 산출 (결정론적, LLM 미사용)
  2) llm_explainer.py : 1)의 수치만 번역하는 자연어 설명 (LLM, 검증 로직 포함)
        │
        ▼
app.py (Streamlit) — 종목 입력 → 예측 + 확률 시각화 + 판단 근거 + 차트
```

### 모델 / 알고리즘 선택

| 레이어 | 선택 | 이유 |
|---|---|---|
| RL 라이브러리 | **stable-baselines3** | 검증된 구현, gymnasium과 바로 호환, 직접 구현 대비 버그 위험 적음 |
| 알고리즘 | **PPO** | 이산 행동(매수/매도/유보)에 안정적, 하이퍼파라미터에 덜 민감해 초기 구축에 적합 |
| 정책망 | **MLP `[64, 64]`로 시작** | state가 이미 SMA/RSI 등으로 가공된 압축 정보라 LSTM의 이점이 제한적. 추후 `policy_kwargs`만 바꿔 LSTM 비교 실험 가능하게 구조화 |

PPO 대안으로 DQN(학습 불안정, replay buffer 필요)이나 SAC(연속 행동에 강함,
지금은 불필요)도 검토했으나, 현재 요구사항(이산 행동 3종, 초기 구축 단계)에는
PPO + MLP가 가장 적합합니다.

## STEP 1. 프로젝트 구조 설계 합의

```
나는 종목별 일자별 주식 시세 데이터를 갖고 있어. 컬럼은:
- 종목코드 (string), 일자 (YYYYMMDD), 시가, 종가, 고가, 저가

이 데이터로 PPO(stable-baselines3) 강화학습 에이전트를 만들어서
"매수 / 매도 / 유보" 3가지 행동을 결정하게 하고 싶어.

핵심 원칙:
- 이 모델은 "2일 후 가격 방향을 맞히는 퀴즈"가 아니라, "지금 포지션과 지표
  상태에서 어떤 행동을 해야 자산가치가 늘어나는가"를 학습하는 매매 정책이야.
- 모델은 하나만 학습하고, 여러 종목의 데이터를 합쳐서 학습 데이터로 사용해.
  단, 지표 계산과 episode(포지션·손익 시뮬레이션) 구성은 반드시 종목별로
  분리해서 계산하고, 종목 경계를 넘어 섞이지 않게 해줘. 학습 단계에서만
  여러 종목의 episode를 섞어서 모델에 입력해.
- state에는 종목코드 자체를 포함하지 않아 — 모델이 지표 패턴의 일반 규칙을
  배우게 하려는 목적이야.

state(9차원, lookback 20일 시퀀스):
SMA5/20/60 이격률, EMA20, 골든크로스 플래그, 데드크로스 플래그,
이격도20(100 중심화), RSI14(정규화), 현재 포지션(보유여부), 미실현손익

reward: 자산가치 변화율 - 거래비용(기본 0.2~0.3%, config화)
  + RSI>70인데 매수 시 페널티(-0.1)
  + 골든크로스 직후 보유 유지 시 보너스(+0.05)
  + 이격도>110인데 신규 매수 시 페널티(-0.1)
  (가중치는 전부 config로 분리, on/off 플래그도 추가)

action: Discrete(3) = 0매도/1유보/2매수

알고리즘: PPO, 정책망은 SB3 기본 MlpPolicy, hidden layer [64,64]로 시작.
추후 LSTM 정책망으로 교체 비교할 수 있도록 policy_kwargs를 분리된 config로 빼줘.

다음 모듈로 나눠서 작업하자:
- data/ : CSV·DB 로딩, 종목코드/종목명 매핑, 종목별 시간 순 정렬
- indicators/ : 기술지표 계산 (종목별 분리 계산)
- env/ : gymnasium TradingEnv (종목별 episode 경계 분리)
- train/ : PPO 학습 — 여러 종목 episode를 섞어서 단일 모델 학습, 체크포인트 저장
- inference/ : 학습된 모델로 임의 종목의 "오늘" 행동 + 확률 예측
- explain/ : 수치 근거 산출(rule_based) + LLM 자연어 설명(llm_explainer), 2단계 분리
- app.py : Streamlit (종목 입력 → 전체 파이프라인 호출 → 결과 표시)

코드 작성 전에 먼저:
1. 폴더/파일 구조
2. 각 모듈의 입출력 인터페이스 (함수 시그니처 수준)
3. train/ 단계에서 "여러 종목 episode를 섞는" 구체적 방식 (예: 종목별로 episode
   리스트를 만든 뒤 shuffle해서 하나의 학습 큐로 합치는 방법 등)
4. 모델(.zip)이 아직 없을 때와 있을 때 Streamlit 앱이 어떻게 다르게 동작해야
   하는지 — 학습은 별도 CLI 스크립트로 분리하고 Streamlit은 추론만 담당할 것
5. 데이터 분할 전략 (시간 기준 train/validation, 종목 내에서 룩어헤드 누설 방지)

을 markdown으로 정리해서 보여줘. 내가 확인하면 순서대로 구현 시작하자.
```

5. docstring + pytest 단위테스트 (RSI는 알려진 예시값으로 검증)

## STEP 3. Gym 환경 (종목별 episode 분리)

```
TradingEnv(gymnasium.Env)를 작성해줘.

state 벡터 (9차원, lookback 20일 시퀀스 → shape (20, 9)):
[SMA5_disp, SMA20_disp, SMA60_disp, golden_flag, dead_flag,
 disparity20_centered, RSI14_norm, position, unrealized_pnl]

- position: 보유=1, 미보유=0
- unrealized_pnl: 보유 중일 때 (현재 종가-매수가)/매수가, 미보유면 0

action: Discrete(3) — 0=매도, 1=유보, 2=매수
- 미보유인데 매도 선택, 보유 중인데 매수 선택 등 무효 행동의 처리 방식을 제안해줘
  (무시 / 작은 페널티 등)

거래 제약:
- 매수는 현금 있을 때만, 매도는 보유 주식 있을 때만 유효
- 거래비용 비율 파라미터화 (기본 0.2~0.3%)

핵심 — episode 경계:
- 환경은 반드시 "단일 종목" 데이터로만 구성되어야 함 (여러 종목을 한 episode
  안에서 이어붙이지 말 것)
- reset() 시 어떤 종목, 어떤 시작 시점으로 episode를 구성할지는 외부(train 모듈)
  에서 주입받는 구조로 설계 (종목 리스트를 섞는 로직은 env가 아니라 train에서 담당)

episode 종료: 데이터 끝 도달 또는 자산이 일정 비율 이하로 하락 시 종료

reset(), step() 구현, render()는 텍스트로 현재 상태 출력.
random policy로 여러 episode 굴려보는 스모크 테스트 작성, 특히 종목이 바뀔 때
포지션/손익이 올바르게 초기화되는지 검증.
```

## STEP 4. Reward 셰이핑

```
reward 함수를 작성해줘:

기본 보상: R_t = (V_t - V_{t-1})/V_{t-1} - 거래비용
 (V_t = 현금 + 보유주식 평가액)

셰이핑 추가:
R_t_final = R_t
  - 0.1 * [RSI>70인데 매수한 경우]
  + 0.05 * [골든크로스 직후 매수 포지션 유지하는 경우]
  - 0.1 * [이격도>110인데 신규 매수한 경우]

요구사항:
- 가중치(0.1, 0.05, 0.1, 거래비용율)는 전부 config로 분리, 튜닝 가능하게
- 셰이핑 on/off 플래그 추가 (순수 수익 기반 reward와 비교 실험용)
- 각 셰이핑 항이 episode 동안 몇 번 발동했는지 카운트 로깅
  (에이전트가 지표 규칙만 흉내내고 있는지 점검용)
```

## 주의사항

- **데이터 누설 방지**: 지표 계산 시 미래 데이터가 섞이지 않는지(`.rolling()`,
  `.shift()` 사용이 올바른지), train/validation이 시간 기준으로 정확히
  분리됐는지 매 단계 검증
- **거래비용 누락 시 백테스트 왜곡**: 수수료/슬리피지 없이 백테스트하면
  수익률이 비현실적으로 좋게 나옴 — STEP 3, 5에 반영됨
- **LLM 설명의 한계**: STEP 7 검증 로직은 숫자 일치 여부만 잡아주며, "논리적으로
  말이 되는 설명인지"까지 보장하지 않음 — 실서비스 전 사람이 직접 몇 건 검토 필요
- **확률값의 의미**: PPO policy의 행동 확률이 실제 신뢰도와 정확히 일치한다는
  보장은 없음(calibration 문제) — 1위/2위 확률 차이가 작을 때는 신호로 과신하지 말 것
- **이 시스템은 투자 자문이 아닙니다.** 과거 데이터 기반 백테스트 성과가 미래
  수익을 보장하지 않으며, 실거래 적용 전 충분한 out-of-sample 검증과 paper
  trading 기간을 거치는 것을 권장합니다.
